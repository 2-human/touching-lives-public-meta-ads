## Deviations from canonical (project-specific — surfaced for a [spec-session])

1. **Base-relative widget-asset resolution (hub/detail wiring).** `review-bootstrap.js` resolves `review-mode.css` / `review-mode.js` relative to **its own `<script src>`**, not the current page, because TL detail pages live at `/details/{slug}.html` while the widget files sit at the mirror root — a bare `review-mode.css` would 404 from a detail page. credo's pages sit next to the widget so it never hit this. **Candidate refinement:** `inert-entry-button.md` / `FEATURE.md` could note that a project serving the widget from a different directory depth than its pages must resolve assets relative to the bootstrap script. (Not changed upstream — [test-session].)
2. **Chrome-root name `.review-floating-pill` / `.review-toggle-btn`** for the entry pill (vs the `.review-pill` example in `css-isolation.md`). Both keep the `.review-` chrome-root prefix, so the isolation contract holds; only the example name differs.

## Wiring

Every generated page (`feed-preview.html` at hub, `details/{slug}.html` per ad) loads two scripts in `<head>`:

```html
<script src="{rel_prefix}review.config.js" defer></script>
<script src="{rel_prefix}review-bootstrap.js" defer></script>
```

Emitted by `work-setup/build-feed-preview.py :: review_widget_scripts(rel_prefix)` — `rel_prefix` empty for the hub, `../` for detail pages. The variant switcher (Image/Video + ratio) on detail pages keeps each variant zone's `data-comment-id` in sync and sets `data-variant-note` (e.g. `"Photo · 9:16"`); `__rwReveal` flips the switcher so a sidebar comment reveals the right variant with dual-highlight, per `variant-anchors.md` / `click-to-reveal.md`.
